umgr

Streamline account lifecycle across platform APIs.

Import current users, detect drift from desired state, and reconcile safely with plan/apply workflows.

What it does

  • Uses YAML or JSON desired-state config files.
  • Tracks current state in local state storage.
  • Imports live users from providers/plugins.
  • Builds actionable drift-aware plans.
  • Applies changes and persists updated state.

Interfaces

  • CLI for operators.
  • Ruby gem API for embedding in larger systems.

CLI quickstart

umgr init
umgr import --config users.yml
umgr validate --config users.yml
umgr plan --config users.yml
umgr apply --config users.yml
umgr show

DSL compile workflow

umgr compile > umgr.yml
umgr validate --config umgr.yml
umgr plan --config umgr.yml
umgr apply --config umgr.yml
umgr import --config umgr.yml
umgr compile | umgr plan --config -
umgr compile | umgr apply --config -

Runtime config precedence

  1. --config <path>: use the explicit config file.
  2. --config -: read config from stdin.
  3. No --config: auto-discover umgr.yml, then umgr.yaml, then umgr.json.

If umgr.rb and a static config file are both present with no explicit --config, umgr fails fast and asks you to use --config or compile pipeline mode.

Gemfile and Ruby API

# Gemfile
gem "umgr"
require "umgr"

runner = Umgr::Runner.new
plan = runner.dispatch(:plan, config: "users.yml")
apply = runner.dispatch(:apply, config: "users.yml")

puts plan[:status]
puts apply[:status]

Provider model

Providers implement platform-specific behavior while keeping a common runtime contract (`validate`, `current`, `plan`, `apply`).
